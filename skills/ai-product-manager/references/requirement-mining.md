## 需求收集方法

### 1. 用户访谈

#### 定性深度访谈
**适用场景**：探索用户深层需求、理解用户行为动机、发现潜在问题

**访谈准备**：
- 明确访谈目标：要解决什么问题？验证什么假设？
- 设计访谈提纲：核心问题（3-5个）、追问问题（10-15个）
- 邀请目标用户：5-10人，覆盖不同用户角色
- 准备录音设备：征得用户同意后录音

**访谈技巧**：
- **开放式提问**："请描述一下您平时是如何处理XX的？"而非"您是否使用过XX功能？"
- **追问式探索**："为什么您会这样操作？"、"您期望的结果是什么？"
- **情景重现**："能举个具体的例子吗？"、"当时您是怎么想的？"
- **沉默留白**：给用户思考时间，不要急于打断

**访谈结构**：
```
开场（5分钟）
├─ 自我介绍
├─ 说明访谈目的和时长
└─ 建立信任关系

背景了解（10分钟）
├─ 了解用户角色和工作场景
├─ 了解现有解决方案
└─ 了解基本使用习惯

核心访谈（30-40分钟）
├─ 探索使用场景
├─ 深挖痛点和需求
└─ 了解期望和偏好

结束（5分钟）
├─ 总结关键发现
└─ 表达感谢和后续联系
```

**访谈记录模板**：
```

- 姓名/ID：
- 角色：
- 工作年限：
- 使用频率：

访谈时间：YYYY-MM-DD HH:MM
访谈地点：线上/线下

关键问题与回答：
Q1: 描述一下您日常工作中如何处理XX？
A1: [用户回答]

Q2: ...
A2: ...

痛点总结：
1. [痛点1] - 严重度：[高/中/低]
2. [痛点2] - 严重度：[高/中/低]

需求洞察：
[从访谈中提取的关键洞察]
```

#### 用户观察
**适用场景**：发现用户真实行为与口头表达的差异、理解用户使用环境

**观察方法**：
- **现场观察**：到用户工作场所，观察实际操作过程
- **录屏观察**：让用户在可控环境下使用产品，录屏分析
- **行为追踪**：通过数据分析工具追踪用户行为路径

**观察要点**：
- 用户操作流程是否顺畅？是否有犹豫、反复操作？
- 用户是否使用预期功能？是否有未预期的使用方式？
- 用户遇到困难时如何解决？是否寻求帮助？
- 用户表情和情绪变化：困惑、烦躁、满意、惊喜

- 直接竞品：功能高度相似
- 间接竞品：解决相似问题的不同方案
- 参考竞品：优秀的设计模式

### 3. 数据分析

#### 用户行为数据分析
**数据来源**：
- 用户行为日志（点击、浏览、操作路径）
- 功能使用数据（使用频率、使用时长）
- 转化漏斗数据（注册→使用→付费）
- 留存数据（次日留存、7日留存、30日留存）

**分析方法**：
- **漏斗分析**：识别流失环节
  ```
  用户进入 → 浏览 → 尝试功能 → 完成操作
  1000人 → 800人 → 300人 → 100人
  转化率: 100% → 80% → 37.5% → 12.5%
  分析：尝试功能到完成操作流失率高，可能原因：功能复杂、操作困难
  ```

- **路径分析**：发现用户真实使用习惯
  ```
  预期路径：首页 → 搜索 → 查看结果 → 购买
  实际路径：首页 → 分类浏览 → 查看详情 → 购买
  洞察：用户更习惯分类浏览而非搜索，需要优化分类设计
  ```

- **分群分析**：对比不同用户群体
  ```
  新用户 vs 老用户：
  - 新用户：首次使用时长短，快速流失
  - 老用户：使用频率高，留存率高
  洞察：需要优化新手引导，降低新用户学习成本
  ```

#### 反馈数据分析
**数据来源**：
- 用户反馈（评分、评论、建议）
- 客服记录（工单、对话记录）
- 社交媒体（微博、论坛、社群）

**分析方法**：
- 情感分析：正负面情感占比
- 主题聚类：按话题归类反馈
- 频次统计：高频问题识别

### 2. 需求分类

#### 按需求性质分类
- **功能需求**：产品需要实现的功能
  - 示例：支持多语言对话、提供代码生成功能
- **非功能需求**：产品的质量属性
  - 性能：响应时间<3秒
  - 安全：数据加密存储
  - 可用性：支持1000并发
  - 兼容性：支持Chrome/Safari/Firefox

#### 按Kano模型分类
- **基本需求**：用户认为理所当然的需求，不满足会不满
  - 示例：客服Agent必须能回答基础问题
- **期望需求**：满足越多越好的需求
  - 示例：响应时间越快越好
- **兴奋需求**：超出用户期望的需求，会带来惊喜
  - 示例：主动识别用户未明确表达的需求
- **反向需求**：提供反而降低满意度
  - 示例：过多的推荐干扰用户决策

#### 按AI特性分类
- **确定性需求**：需求明确，输入输出确定
  - 示例：数据格式转换
- **探索性需求**：需求有一定灵活性，AI可发挥创意
  - 示例：文案创作
- **模糊需求**：需求不清晰，需要AI理解意图
  - 示例："帮我优化这个"

## 需求优先级评估

### 1. 评估维度

| 维度 | 说明 | 评分（1-5分） |
|-----|------|-------------|
| **用户价值** | 对用户的价值大小 | 5=极高，1=低 |
| **商业价值** | 对业务的贡献大小 | 5=极高，1=低 |
| **实现难度** | 技术实现难度 | 5=极难，1=容易 |
| **资源需求** | 需要的人力、时间、成本 | 5=极高，1=低 |
| **依赖关系** | 是否依赖其他功能 | 5=强依赖，1=无依赖 |
| **紧迫程度** | 是否需要立即实现 | 5=非常紧急，1=不紧急 |

### 2. 优先级矩阵

```
        高用户价值
           │
   ┌───────┼───────┐
   │ P0    │ P1    │
   │ 立即做│ 尽快做│
   ├───────┼───────┤
   │ P2    │ P3    │
   │ 计划做│ 暂不做│
   └───────┼───────┘
        低用户价值
   高实现成本 ←───────→ 低实现成本

P0（立即做）：高价值+低成本
P1（尽快做）：高价值+中成本 或 中价值+低成本
P2（计划做）：中价值+中成本
P3（暂不做）：低价值 或 高成本
```

### 3. 优先级排序方法

#### 加权评分法
```
综合得分 = Σ（维度评分 × 权重）

权重配置：
- 用户价值：30%
- 商业价值：20%
- 实现难度：-20%（负权重）
- 资源需求：-15%（负权重）
- 紧迫程度：15%
- 依赖关系：-10%（负权重）

示例：
需求A：用户价值5，商业价值4，实现难度3，资源需求2，紧迫程度4，依赖关系2
综合得分 = 5×30% + 4×20% - 3×20% - 2×15% + 4×15% - 2×10%
        = 1.5 + 0.8 - 0.6 - 0.3 + 0.6 - 0.2
        = 1.8
```

#### MoSCoW法
- **Must have**（必须有）：MVP核心功能，不做无法上线
- **Should have**（应该有）：重要但可以延后
- **Could have**（可以有）：锦上添花的功能
- **Won't have**（不会有）：明确不做

#### 四象限法
```
紧急程度
高
│  ┌────────┬────────┐
│  │ 重要紧急│ 重要不紧急│
│  │   Q1   │   Q2   │
│  │ 立即做 │ 计划做 │
低│  ├────────┼────────┤
│  │ 不重要紧急│不重要不紧急│
│  │   Q3   │   Q4   │
│  │ 授权/外包│ 不做   │
│  └────────┴────────┘
│     重要        不重要
```

### 4. MVP功能清单

**MVP（Minimum Viable Product）最小可行产品**：用最少的资源验证核心假设。

**MVP功能选择原则**：
1. **用户价值优先**：解决最核心的用户痛点
2. **技术可行性**：能够在短期内实现
3. **数据验证导向**：能够验证关键假设
4. **低成本试错**：失败成本可控

**MVP功能清单示例**（智能客服Agent）：

| 功能 | 用户价值 | 技术难度 | 是否MVP | 理由 |
|-----|---------|---------|---------|------|
| FAQ自动回答 | 高 | 低 | ✓ | 核心功能，解决高频咨询 |
| 订单查询 | 高 | 中 | ✓ | 核心功能，高价值场景 |
| 投诉处理 | 中 | 高 | ✗ | 复杂度高，后续版本 |
| 产品推荐 | 中 | 中 | ✗ | 非核心功能 |
| 知识库管理 | 中 | 中 | ✓ | 必要支撑功能 |
| 数据分析 | 低 | 中 | ✗ | 后续版本 |

## 常见误区

### 误区1：混淆用户需求与用户行为
**问题**：直接把用户要求的功能当作需求，没有深挖背后动机。

**正确做法**：
- 用户说"我想要一个导出功能" → 深挖"你为什么需要导出？" → "我想分享给同事" → 实际需求是"协作分享"，可能有更好的方案

### 误区2：忽视用户真实能力
**问题**：假设用户具备较高的技术能力或理解能力。

**正确做法**：
- 分析目标用户的技术水平
- 设计符合用户能力的交互方式
- 提供清晰的引导和帮助

### 误区3：过度追求完美
**问题**：试图一次性满足所有需求，导致项目延期。

**正确做法**：
- 采用MVP思维，先实现核心功能
- 快速迭代，根据反馈调整

### 误区4：忽视数据验证
**问题**：仅凭访谈和主观判断决定需求。

**正确做法**：
- 结合数据分析验证需求
- 通过A/B测试验证效果

### 误区5：忽视非功能需求
**问题**：只关注功能需求，忽视性能、安全等非功能需求。

**正确做法**：
- 明确性能指标（响应时间、并发数）
- 定义安全要求（数据加密、权限控制）
- 规划可用性保障（容灾、备份）

## 示例

### 示例1：代码助手需求挖掘

**用户访谈发现**：

### 示例2：内容创作平台需求分析

**数据洞察**：
- 用户留存率：首日40%，7日15%，30日5% → 问题：用户粘性低
- 功能使用率：文案生成80%，图片生成60%，视频生成10% → 机会：视频生成用户少，但需求大
- 用户反馈：60%用户反馈"生成内容质量不稳定" → 痛点：质量不可控

**优先级评估**：

| 功能 | 用户价值 | 实现难度 | 综合得分 | 优先级 |
|-----|---------|---------|---------|--------|
| 提升内容质量稳定性 | 5 | 4 | 1.4 | P0 |
| 新增视频生成 | 4 | 5 | 0.8 | P1 |
| 优化用户体验 | 3 | 3 | 1.2 | P1 |
| 新增模板库 | 2 | 2 | 1.0 | P2 |

**决策**：
- P0：立即优化内容质量稳定性（核心痛点）
- P1：优化用户体验提升留存、规划视频生成（高价值）
- P2：新增模板库（增值功能）
